# Neovim 0.12 Enhancements — Design

**Date:** 2026-04-17
**Scope:** `/Users/hohung/.config/nvim`
**Target Neovim version:** 0.12.1

## Goal

Update the existing Neovim config to (a) stay compatible with 0.12 by migrating deprecated APIs, and (b) adopt the highest-value new features from 0.12 without disturbing existing workflow, keymaps, or plugin set.

This is "Approach 2 — Balanced" from brainstorming. Approach 3 (migrating off `lsp-zero` / `mason-lspconfig v1` handlers / enabling experimental `ui2`) is explicitly out of scope.

## Non-goals

- Replacing `nvim-cmp` with `blink.cmp`.
- Removing `lsp-zero` or migrating to native `vim.lsp.config()` / `vim.lsp.enable()`.
- Upgrading `mason-lspconfig` from v1 `handlers{}` API to v2.
- Enabling experimental `ui2`.
- Adding LSP `textDocument/inlineCompletion` (overlaps with `copilot.lua` ghost text).

## Context

Current setup:
- `folke/lazy.nvim` on `main` branch — compatible with 0.12, no migration needed.
- LSP stack: `nvim-lspconfig` + `mason.nvim` + `mason-lspconfig` (v1 `handlers{}`) + `lsp-zero v3.x`.
- Completion: `nvim-cmp` with `cmp-nvim-lsp`, `cmp-buffer`, `cmp-path`, `LuaSnip`.
- Inline AI completion: `zbirenbaum/copilot.lua` with ghost text on `<C-a>`.
- Other: `telescope`, `oil`, `bufferline`, `lualine`, `gitsigns`, `lspsaga`, `lazygit`, `indent-blankline`, `nvim-dap`, `smear_cursor`, `vim-visual-multi`, `git-conflict`.

Deprecated APIs currently in use:
- `vim.diagnostic.goto_next` / `vim.diagnostic.goto_prev` (in `lua/options.lua` lines 40–41).
- `vim.lsp.util.make_position_params()` with no explicit `offset_encoding` (in `lua/plugins/lsp.lua` Rust `ExpandMacro` command).
- `vim.lsp.util.make_range_params()` with no explicit `offset_encoding` (in `lua/plugins/lsp.lua` Go `BufWritePre` autocmd).

## Design

### Section 1 — Deprecation fixes

Replace deprecated API calls with their 0.12 equivalents. Keymaps and observable behavior remain identical.

| File:Line | From | To |
|---|---|---|
| `lua/options.lua:40` | `vim.diagnostic.goto_next` | `function() vim.diagnostic.jump({ count = 1, float = true }) end` |
| `lua/options.lua:41` | `vim.diagnostic.goto_prev` | `function() vim.diagnostic.jump({ count = -1, float = true }) end` |
| `lua/plugins/lsp.lua` (Rust `ExpandMacro`) | `vim.lsp.util.make_position_params()` | `vim.lsp.util.make_position_params(0, client.offset_encoding)` — pass explicit encoding from the requesting client |
| `lua/plugins/lsp.lua` (Go `BufWritePre`) | `vim.lsp.util.make_range_params()` | `vim.lsp.util.make_range_params(0, client.offset_encoding)` — pass explicit encoding |

For the Go `BufWritePre` autocmd, the current code calls `make_range_params()` before it knows which client it's sending to. In the fix, we resolve the first LSP client attached to the buffer and use its `offset_encoding`. Fallback: `"utf-16"` (the spec default) if no client is attached — though the autocmd's subsequent `buf_request_sync` wouldn't do anything meaningful in that case anyway.

For the Rust `ExpandMacro` command, `make_position_params()` is called outside a client context; we look up the `rust_analyzer` client by name (it's the one that implements `rust-analyzer/expandMacro`) and use its `offset_encoding`.

### Section 2 — New 0.12 features

All additive. No existing keymap or behavior is removed or changed.

#### 2a. Smarter diagnostic jumps

Keep `]d`/`[d` mapped to jump to the next/prev diagnostic of any severity (behavior unchanged). Add four new optional bindings that leverage `vim.diagnostic.jump`'s `severity` option:

| Key | Action |
|---|---|
| `]e` | Jump to next error (`severity = ERROR`) |
| `[e` | Jump to prev error |
| `]w` | Jump to next warning (`severity = WARN`) |
| `[w` | Jump to prev warning |

All four open a float on arrival (`float = true`).

#### 2b. Linked editing range (LSP)

When a language server implements `textDocument/linkedEditingRange`, Neovim 0.12 can auto-sync edits across occurrences in the current scope (e.g. renaming an HTML/JSX tag updates the closing tag live).

Implementation:
- Inside a shared `on_attach`, if `client.supports_method("textDocument/linkedEditingRange")`, call `vim.lsp.linked_editing_range.enable(true, { client_id = client.id, bufnr = bufnr })`.
- Optional toggle keymap: `<leader>lL` (capital L, doesn't collide with `<leader>lr`/`lf`/`la`/`le`) toggles the feature per-buffer.

#### 2c. Inlay hints toggle

`vim.lsp.inlay_hint` matured in 0.12.

Implementation:
- In `on_attach`, if `client.supports_method("textDocument/inlayHint")`, call `vim.lsp.inlay_hint.enable(true, { bufnr = bufnr })`.
- Add `<leader>lh` to toggle inlay hints on the current buffer: `vim.lsp.inlay_hint.enable(not vim.lsp.inlay_hint.is_enabled({ bufnr = 0 }), { bufnr = 0 })`.

#### 2d. `:lsp` command

Built-in in 0.12 for interactive LSP client management (list/info/start/stop/restart attached clients — exact subcommands to be confirmed at implementation time via `:help lsp-command`). No configuration required. Nothing to add to config; simply document in `Keymaps.md` so it's discoverable.

#### 2e. Native TUI progress

OSC 9;4 progress sequences are emitted automatically by Neovim 0.12 for long-running operations. No config change. Works if the host terminal supports it (recent WezTerm, Kitty, iTerm2); silently ignored otherwise.

### Section 3 — Shared `on_attach` helper

To avoid duplicating logic inside each server's setup block, introduce a single local `on_attach` function inside `lua/plugins/lsp.lua` that:

1. Enables linked editing if supported (2b).
2. Enables inlay hints if supported (2c).
3. Sets buffer-local `<leader>lh` and `<leader>lL` keymaps.

Wire `on_attach = on_attach` into each `require("lspconfig")[server_name].setup({...})` call in the `handlers` table (three places: `rust_analyzer`, `sqls`, default fallback).

The existing `sqls` `on_attach` (which calls `require("sqls").on_attach` and sets `omnifunc`) must be preserved — the shared helper is called *in addition*.

## Rollout plan

Executed in this order, with verification after each step:

1. **Snapshot state** — commit any pending changes; current `lazy-lock.json` is the rollback anchor.
2. **`:Lazy sync`** — pull 0.12-compatible plugin updates. Lock file will change.
3. **Apply Section 1 (deprecation fixes)** — restart nvim.
   - Verify: `]d`/`[d` work; `.go` save still organizes imports + formats; `:ExpandMacro` in a Rust file still splits and shows expansion.
4. **Apply Section 2a (new diagnostic jump keys)** — verify `]e`/`[e`/`]w`/`[w` behave correctly on a file with mixed-severity diagnostics.
5. **Apply Section 2b + 2c + 3 (shared `on_attach`)** — verify on a Go or Rust file: inlay hints appear and `<leader>lh` toggles them; linked editing toggle works on a server that supports it (or fails silently otherwise).
6. **Document `:lsp` in `Keymaps.md`** (Section 2d).

## Verification checklist

- `:checkhealth` reports no new errors after changes.
- `]d` / `[d` still jump to next/prev diagnostic of any severity.
- `]e` / `[e` jump only to errors; `]w` / `[w` only to warnings.
- Saving a `.go` file still organizes imports and formats (no regressions).
- Saving a `.sql` file still formats.
- `:ExpandMacro` in a Rust file still opens a vertical split with the expansion.
- `<leader>lh` toggles inlay hints on the current buffer.
- `<leader>lL` toggles linked editing range.
- `:lsp` command is recognized and lists/manages attached clients.

## Rollback

If any step produces unexpected behavior:

- Code rollback: `git checkout lua/options.lua lua/plugins/lsp.lua docs/superpowers/specs/2026-04-17-nvim-0.12-enhancements-design.md`
- Plugin rollback: `git checkout lazy-lock.json && :Lazy restore` restores the previous plugin versions.

## File change summary

- `lua/options.lua` — replace two keymap callbacks; add four new keymaps for severity-filtered jumps.
- `lua/plugins/lsp.lua` — add shared `on_attach` helper; wire it into three server setups; pass explicit `offset_encoding` to `make_position_params`/`make_range_params` calls.
- `Keymaps.md` — document new `]e`/`[e`/`]w`/`[w`/`<leader>lh`/`<leader>lL` mappings and the `:lsp` command.
- `lazy-lock.json` — will update naturally from `:Lazy sync`.

No new plugins. No files deleted.
